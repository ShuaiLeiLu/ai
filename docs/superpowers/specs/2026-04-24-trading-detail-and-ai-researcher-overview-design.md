# 模拟盘详情与 AI 研究员总览改版设计

## 背景

当前系统已经具备模拟盘、研究员工作台、热门文档、公开排行等基础能力，但存在三个明显问题：

1. 模拟盘详情的数据口径不稳定，用户对总资产、持仓盈亏、收益率、历史收益的信任感不足。
2. 模拟盘详情请求过重，首屏容易被历史交易、图表统计、外部行情刷新拖慢。
3. AI 研究员总览虽然已有页面骨架，但还没有成为真正的主入口，页面结构和信息优先级弱于对标产品。

本次改版目标是在一个实现周期内完成两件事：

1. 先将模拟盘详情改造成可信、快速、可阅读的交易驾驶舱。
2. 再将 AI 研究员总览改造成工作台主入口，承接研究员、文档、模拟盘排行三个核心场景。

## 范围

本次只覆盖以下两块：

1. 模拟盘详情页
2. AI 研究员总览页

本次明确不做：

1. 题材掘金新页面
2. 资讯分析深改
3. 社区能力扩展
4. 完整 RQAlpha 引擎接入

## 目标

### 目标一：模拟盘详情可信

用户打开模拟盘详情时，应能直接理解并信任：

1. 当前账户总资产是多少
2. 每个持仓赚亏多少
3. 今日盈亏和累计收益率是多少
4. 历史交易和收益图表为什么得出当前结果
5. 每次交易后的 AI 复盘与次日展望是什么

### 目标二：模拟盘详情足够快

用户打开详情页时，首屏只加载必要数据，历史图表和明细在用户真正查看时再请求，不允许同步依赖外部行情接口。

### 目标三：AI 研究员总览成为主入口

总览页应首先回答三个问题：

1. 当前有哪些研究员在工作
2. 最近产出了什么内容
3. 谁的模拟盘表现最好

## 设计一：模拟盘详情

### 页面结构

页面拆成四个区块：

1. 顶部账户概览
2. 左侧持仓与最近交易侧栏
3. 右侧交易日志主视图
4. 右侧历史交易视图

#### 顶部账户概览

展示以下指标：

1. 总资产
2. 今日盈亏
3. 累计收益率
4. 持仓市值
5. 可用资金

这些指标必须全部来自同一份账户快照，避免不同接口分别计算后出现口径漂移。

#### 左侧持仓与最近交易侧栏

保留“当前持仓 / 最近交易”切换，但职责简化：

1. 当前持仓只展示持仓摘要，按持仓盈亏排序
2. 最近交易只展示轻量摘要，不在侧栏中做复杂统计

#### 右侧交易日志主视图

默认主 tab 为交易日志，承担主要解释功能。每次交易日志展示：

1. 成交明细
2. 交易复盘
3. 执行反思
4. 次日展望

这部分作为详情页默认阅读区，优先级高于历史图表。

#### 右侧历史交易视图

历史交易 tab 展示：

1. 收益曲线
2. 月度收益
3. 风险指标
4. 投资日历
5. 成交明细表

这部分不参与首屏阻塞，按需加载。

### 数据口径

#### 账户快照口径

账户摘要必须统一来源于持久化快照字段，包括：

1. `total_asset`
2. `available_cash`
3. `holding_value`
4. `daily_pnl`

页面读接口默认不在请求链路中同步打外部行情，而是使用已落库快照或缓存快照。

#### 持仓口径

持仓数据展示：

1. 成本价
2. 当前价
3. 持仓盈亏
4. 盈亏比例

当前价默认从 Redis 行情缓存或数据库快照读取，不在详情页请求内触发行情源调用。

#### 历史收益口径

收益曲线、月度收益、风险指标、投资日历必须基于统一成交回放逻辑生成，不能一部分来自账户表、一部分来自临时估算。

### 请求策略

详情页首屏只请求：

1. `account`
2. `positions`
3. `latest logs`

以下数据按需请求：

1. `records`
2. `stats`
3. `history charts`

触发条件：

1. 用户切换到历史交易 tab 后再拉取历史数据
2. 用户切换到交易日志 tab 时只拉取日志，不强制拉全量历史记录

### 性能策略

1. 页面只读 DB/Redis 快照，不同步请求外部行情
2. 账户映射、账户摘要、持仓摘要保留短缓存
3. 历史统计与成交回放结果可缓存，避免重复计算
4. 页面禁止请求“模拟盘详情全量大包”接口，改为按功能拆分

### AI 解读展示

每次交易后的 AI 内容统一成固定结构：

1. 交易复盘
2. 执行反思
3. 次日展望

展示上要求：

1. 日志结构一致
2. 模型失败时仍有模板化兜底文本
3. 买入和卖出日志都走同一套阅读框架

## 设计二：AI 研究员总览

### 页面定位

AI 研究员总览不再只是“研究员列表页”，而是整个工作台的主入口页。

### 页面结构

页面分为左中两大块：

1. 左侧研究员面板
2. 中间主内容区

#### 左侧研究员面板

包含：

1. `AI研究员` 标题与说明
2. 固定的 `总览` 入口
3. 已雇佣研究员卡片列表
4. `招募研究员` 与 `创建研究员` 入口

研究员卡片重点展示：

1. 头像
2. 名称
3. 当前状态

#### 中间主内容区

在“总览”状态下展示：

1. 热门文档
2. 公开模拟盘排行

在“研究员详情”状态下展示：

1. 研究员 header
2. 最新制品
3. 模拟盘摘要卡

这里的模拟盘只展示轻摘要，不直接内嵌完整详情页。

### 数据策略

总览页只读取轻量摘要接口，不请求模拟盘详情的完整数据包。

总览页最小数据集包括：

1. 已雇佣研究员列表
2. 热门文档列表
3. 公开模拟盘排行
4. 单研究员模拟盘轻量摘要

### 交互策略

1. 默认进入总览
2. 点击研究员后切换到研究员详情视图
3. 点击模拟盘摘要后进入模拟盘详情页
4. 首页不展示历史交易、图表统计、日志长列表

## 实现顺序

### 第一阶段：模拟盘详情

1. 收口账户快照、持仓、收益统计口径
2. 调整详情页请求边界，按需加载历史数据
3. 重构详情页排版层级
4. 完善 AI 解读展示

### 第二阶段：AI 研究员总览

1. 抽取总览页需要的轻量模拟盘摘要数据
2. 重构左侧研究员导航与总览态
3. 重构热门文档与公开模拟盘排行展示
4. 打通总览与详情页的跳转关系

## 错误处理

1. Redis 无缓存时使用数据库快照兜底
2. 外部行情源异常不能阻塞页面读请求
3. AI 解读失败时使用模板兜底，不留空白日志
4. 历史数据为空时要有清晰空状态，不出现错位或假数据

## 测试与验收

### 模拟盘详情

验收标准：

1. 首屏打开明显快于当前版本
2. 总资产、持仓盈亏、收益率口径一致
3. 切换历史交易时不会拖慢首屏
4. AI 复盘可正常展示

### AI 研究员总览

验收标准：

1. 页面成为真正的主入口
2. 研究员、热门文档、模拟盘排行三块形成闭环
3. 总览页不再请求模拟盘详情重数据
4. 点击研究员与点击模拟盘详情的路径清晰

## 风险与控制

1. 如果模拟盘收益口径仍然分散在多个接口，需优先统一后再做视觉层改动。
2. 如果总览页继续复用详情重接口，会导致首页变慢，必须禁止。
3. 如果历史统计完全实时计算，页面性能会再次恶化，因此需要缓存和按需加载一起落地。
